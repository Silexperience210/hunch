    **`docs/HIP-5.md` (PROTO-06) — Reputation Event Format and Aggregation Rules**

    Sections:
    - Abstract: defines kind:30891 Nostr reputation event format + aggregation rules for Hunch oracles, mints, and market creators. Time-decayed weighting + social-graph-aware sum.
    - Motivation: Hunch uses **reputation, not slashing** (PROJECT.md key decision — Bitcoin has no smart contract to slash). Reputation must be cryptographically signed (Nostr), aggregable by frontends, gameable-resistant (Sybil resistance via social graph), and time-decayed (a 2-year-old vouch weighs less than a 2-month-old one).
    - Specification:
      - Event format: kind:30891 (parameterized replaceable, addressable), `d` tag = target npub. Tags: `target` (npub of oracle/mint/creator), `target_type` (oracle | mint | creator), `vote` (vouch | report), `weight` (0-100 int — voter's confidence). Optional `evidence` (URL or event ID). Content: optional free-form note.
      - Replaceability: each (voter, target) pair has at most one current reputation event. Voter updates by publishing a new event with the same `d` tag.
      - Aggregation rules (frontend-side):
        - Raw score: `Σ (weight × decay_factor × graph_distance_factor) for all vouch events` − `Σ (weight × decay_factor × graph_distance_factor) for all report events`.
        - `decay_factor = exp(-age_days / 180)` (half-life ~125 days, full-decay ~2 years).
        - `graph_distance_factor`: 1.0 for 1-hop, 0.5 for 2-hop, 0.25 for 3-hop, 0 for beyond. Sybil-resistant.
        - Normalized score: `raw / Σ(theoretical_max_weight)` × 100, clamped [-100, 100].
      - Minimum threshold to display: a target with <5 distinct voters within the viewer's 2-hop social graph is shown as "Insufficient reputation data" — protects against zero-data manipulation.
    - Rationale: time decay + social-graph weighting + per-voter replaceability discourages buying old accounts to vouch en masse (Augur lesson). Frontend-side aggregation means no centralized reputation registry. Voters' decisions are public per-event but private in aggregate (Nostr's identity model).

    - Backwards Compatibility: rules may tune (decay constant, graph distance falloff). All tuning published as HIP-5a/b/c addenda. Underlying event format is stable.
    - Security Considerations: Sybil farms (mitigated by social-graph distance factor); coordinated brigades (mitigated by graph + 5-voter minimum + community report flags via kind:30890 dispute); reputation poisoning (target reports the brigader; brigader's votes self-discount over time).
